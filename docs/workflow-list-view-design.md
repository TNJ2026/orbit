# Workflow 列表视图设计（大纲式流程展示）

> 文档状态：Design Draft
> 定位：[workflow-prompt-authoring-design.md](workflow-prompt-authoring-design.md) 的姊妹篇——那份文档解决"图从哪来"（提示词生成），本文档解决"流程怎么看、怎么改"（默认视图从画布换成大纲列表）。
> 引擎模型零改动：节点类型、outcome port、校验、revision 全部照用（[general-workflow-design.md](general-workflow-design.md)）；本文只改展示与编辑入口。

## 1. 背景与动机

### 1.1 问题

现有 Workflow 页以图画布为唯一视图。用户反馈的复杂度大部分来自图本身：

- 二维自由度是负担：坐标、连线、锚点、布局都要用户管理，而流程信息的本质是**顺序 + 例外**；
- rework 回边是把图画乱的元凶：一条"驳回退回起草"在画布上是一根跨越多节点的弧线，交叉、重叠、难以点选；
- 运行时找状态要扫全画布："现在卡在哪一步"没有自然的阅读顺序；
- 双击弹窗编辑打断上下文：改一个步骤的 prompt 时看不见前后步骤。

### 1.2 观察

orbit 的工作流形态 90% 是**线性主干 + 返工回边 + 少量并行**。这种结构和人描述流程的方式同构（"先…然后…如果不行就退回…"），也和提示词生成的输入形态同构。垂直列表让输入（自然语言描述）、输出（生成的流程）、阅读（确认与运行监控）共享同一个心智模型。

先例：Zapier（线性步骤列表）、Apple 快捷指令（嵌套块列表，含条件和循环）、GitHub Actions（job 列表 + 依赖）。它们都证明：面向非图论用户的流程产品，列表是默认视图，图是高级视图。

### 1.3 原则

1. **列表为默认视图，画布降级为"高级视图"切换**——不删画布，现有代码零沉没成本；
2. **顺序优先**：主干从上到下一遍读完；例外（回跳、分支）以标注形式附着在主干上，不打断阅读流；
3. **行是操作台不是文档**：行内只放扫读所需的最少信息，详情进侧栏；
4. **数据层零改动**：渲染数据源就是现有 `read_workflow_config` 的 cfg（steps/edges/subflows），不新增存储结构。

## 2. 视图结构总览

```text
┌─ Workflow: 内容发布 ────────────────────────── [列表|画布] ─┐
│                                                              │
│  1 ▢ 起草                        agent: writer               │
│      └ "按 brief 起草初稿,产出存 docs/draft.md…"      ✎     │
│  2 ⇉ 并行 ── 全部通过才继续 (all)                            │
│      ├ ▢ 编辑审核                agent: editor    ✋审批     │
│      └ ▢ 法务审核                agent: legal     ✋审批     │
│          ↩ 驳回 → 回到第 1 步    (最多 3 次)                 │
│  3 ◇ 质量分 ≥ 80 ?                                           │
│      ├ 是 → 继续                                             │
│      └ 否 → 回到第 1 步                                      │
│  4 ▣ 逐项处理 (×N 并发 4)                    [展开 ▾]        │
│  5 ▢ 发布                        agent: publisher  🔒串行    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
                          ┌─ 侧栏：选中步骤详情（§5）─┐
```

页面骨架：左侧主列表（编号大纲），右侧详情面板（选中行的字段编辑），顶部视图切换 `[列表|画布]` 与提示词输入框（authoring 文档 §5.1 的同一个输入框）。

## 3. 图结构 → 列表形式的完整映射

| 图结构 | 列表形式 | 说明 |
|---|---|---|
| 顺序（前向边链） | 编号行，从上到下 | 主干编号 1、2、3…；编号即对话修改的指代锚点（"第 3 步后面加一步"） |
| 并行分支 + join | 缩进块，块头 `⇉ 并行 ── <join 策略>` | join 不占独立行，策略（all / any / quorum(n) / all_successful）显示为块属性；`join_remaining: cancel` 显示为 "达成后取消剩余" |
| decision | `◇ <条件描述>` 行 + 缩进分支 | JSONLogic 条件渲染成人话（`{">=":[{"var":"score"},80]}` → "score ≥ 80"）；每分支一行，目标是"继续"或"回到第 N 步" |
| rework 回边 | `↩ <端口名> → 回到第 N 步 (最多 M 次)` 文字标注 | **不画线**。附着在回边源步骤下方；`max_iterations` 内联显示 |
| 自定义端口出边 | 源步骤下缩进的端口行 `── approved → 第 N 步` | 只列自定义端口；默认 success 主干顺序已表达，不重复 |
| foreach（item DAG） | `▣ 逐项处理 (×N 并发 K)` 可折叠块 | 展开显示 body 子流程列表；item 间 depends_on 在展开视图内用"等待 item X"标注 |
| subflow | `⧉ <名称>` 可折叠块 | 展开内嵌子图的列表（只读预览 + "编辑子流程"跳转）；嵌套编号 4.1、4.2… |
| approval（附着式） | 行内 `✋审批` 徽章 | 与附着式审批设计一致（authoring 文档 §26 决策），不占独立行 |
| approval（独立节点） | `✋ <名称>` 独立行 + 端口分支缩进 | 多级审批链场景 |
| 入口 / end | 隐式 | 第 1 行即入口；主干最后一行即终点。显式 end 节点渲染为 `◉ 结束` 行 |
| `exclusive` 互斥 | 行内 `🔒串行` 徽章 | 悬停显示互斥键 |
| `environment`（worktree 等） | 行内小字徽章（如 `⎇ 隔离工作区`） | 默认折叠进详情面板，仅非默认值时行内提示 |
| verify 命令 | 行内 `✓verify` 徽章 | 命令本体在详情面板 |

### 3.1 缩进语义

只有两种缩进：**并行块**（同级分支并列）和**折叠块**（foreach/subflow 的 body）。条件分支缩进一层但不再嵌套——分支内如果还有复杂结构，引导用户拆 subflow。这条限制保证列表最多三层缩进，超出即提示"结构过于复杂，建议拆分子流程或切画布视图"。

### 3.2 编号规则

- 主干顺序编号（拓扑序，从入口沿前向边）；并行块内用 2a、2b；foreach/subflow 展开用 4.1、4.2；
- 编号是**显示序号**，不进数据层（step.id 不变）；步骤增删后重排；
- 回跳标注引用编号（"回到第 1 步"），编号变化时标注同步重算——数据层回边始终是 `{from, to, port}`，渲染时才翻译成编号。

## 4. 行内信息密度规范

每行从左到右：

```text
<编号> <类型图标> <步骤名>    <agent 徽章> <✋审批> <🔒串行> <✓verify> <状态点>
  └ <prompt 首行摘要,截断 60 字,灰色>                                    <✎>
```

- **必显**：编号、类型图标（▢ action / ◇ decision / ⇉ 并行块头 / ▣ foreach / ⧉ subflow / ✋ approval / ◉ end）、步骤名、agent（未绑定显示红色"未选 Agent"——启动门槛可视化）；
- **有则显**：审批/串行/verify 徽章、自定义端口行、回跳标注；
- **prompt 摘要行**：有 prompt 时显示首行截断（可全局开关）；无 prompt 显示弱提示"未设置提示词"。摘要让 §5.3（authoring 文档）模糊条件写进 prompt 的判断标准可见；
- **不显**：端口默认值（success/rework）、坐标、schema、命令——全部进详情面板。

## 5. 详情面板（含提示词编辑）

点击行 → 右侧面板（替代现有双击弹窗）。**数据层不变：提示词还是 `step.prompt` 字段，存 workflow.json**；面板只是编辑入口。

```text
┌─ 步骤 1:起草 ────────────────────────┐
│ 名称     [起草              ]         │
│ Agent    [writer ▾] [+]              │
│ 提示词                                │
│ ┌──────────────────────────────────┐ │
│ │ 按 brief 起草初稿…                │ │
│ │ (textarea,面板主体,占大半高度)     │ │
│ └──────────────────────────────────┘ │
│ ▸ 受保护约束（只读）                  │
│ ✋ 完成需人工批准        [☐]          │
│ ▸ 高级:端口 / verify / 环境 / 超时    │
└───────────────────────────────────────┘
```

- **面板不遮列表**：改 prompt 时能看见上下文步骤；选中行高亮，↑/↓ 切换步骤连续编辑；
- **prompt 是面板主体**：textarea 占大半高度——它是用户最常改的字段；
- **引擎契约只读展示**：`contract`（review 只读、git_merge 等受保护约束）折叠区展示、不可编辑——用户 prompt 是补充，不能覆盖协议（general-workflow-design §3.1 边界）；
- **高级字段折叠**：端口、verify、环境、超时、schema 收进"高级"，默认不展开；
- 边属性（条件、循环上限、映射）：点击列表里的端口行/回跳标注 → 同一面板切换为边详情。

## 6. 运行时状态视图

同一列表在 Goal 运行时叠加状态层，这是列表相对画布收益最大的场景：

```text
1 ✓ 起草              writer     2m 前完成
2 ⇉ 并行
   ├ ✓ 编辑审核       editor     已批准
   └ ● 法务审核       legal      运行中 3m…
3 ○ 质量门                        待进行
4 ⚠ 发布                          blocked: 等待确认 (needs_confirmation)
```

- 状态点：`✓ 完成 / ● 运行中 / ○ 待进行 / ⚠ blocked / ✋ 等待审批 / ⏸ 冻结`——数据源是现有 transitions 投影（`active_step_assignees` + `workflow_derived_task_status`），node_runs 表可提供 attempt/耗时增强；
- 回跳发生时在主干行下追加历史标注：`↩ 第 2 次返工 (2/3)`——数据源是 rework transitions 计数；
- 点击运行中/blocked 行 → 详情面板切换为运行详情（当前 agent、耗时、最近输出摘要、blocked 原因、`[批准] [请求修改] [重跑] [确认结果]` 操作按钮——对接现有审批/rerun/confirm API）；
- foreach 块头显示进度 `▣ 逐项处理 (3/7 完成)`，展开看单 item 状态。

## 7. Diff 预览（对话式修改的确认视图）

列表行级 diff 比画布着色更接近代码 review 的成熟范式：

- 新增行：绿底；删除行：红底删除线（保留显示，不消失）；字段变化：黄底 + 展开显示 `字段: 旧值 → 新值`；
- prompt 变化：摘要行标黄，点开面板看全文 diff（行级对比）；
- 结构变化（并行改串行、回跳目标变化）：块头/标注标黄；
- 顶部确认条沿用 authoring 文档 §6：`[应用] [放弃] + 摘要（+2 步骤 / -1 边 / 1 处 prompt 修改）`。

## 8. 对话式修改的衔接

列表让 authoring 文档 §7 的对话修改指代更自然：

- **编号即锚点**："第 3 步后面加一步 SEO 检查"、"把第 2 步的并行改成先后"——生成 prompt 附带"当前流程的编号列表文本"（就是本视图的文本序列化），模型引用编号比引用坐标/节点 ID 可靠；
- **列表文本 = 流程的规范文本形式**：`视图渲染(cfg)` 产出的纯文本大纲可直接作为生成端点的 `current` 上下文摘要（替代或补充全量 JSON），token 更省、歧义更少；
- 反向同理：Explain（authoring 文档 §18）的 `steps`/`exceptions` 结构就是本视图的数据结构——**列表视图 = Explain 的确定性渲染 + 交互**，一份实现两用。

## 9. 与画布的关系

- 顶部 `[列表|画布]` 切换，同一份 cfg 两种渲染;选中状态跨视图保持（列表选中第 3 步 → 切画布高亮对应节点）；
- 画布定位为高级视图，两种场景不可替代：多个交叉 join 的非平面图（缩进表达不了交叉依赖）、用户想看拓扑全貌；
- 列表检测到超出表达能力的结构（交叉 join、>3 层嵌套、多入口）时，对应区域显示"此段结构复杂，[在画布中查看]"局部降级，而不是整页拒绝；
- 默认视图记忆用户选择;新用户默认列表。

## 10. 实现要点

### 10.1 渲染算法

1. `read_workflow_config` 取 cfg → `workflow_graph` 取回边集合 → 前向图拓扑排序得主干序；
2. 并行块识别：同一源的多条前向出边 + 汇合点（join 节点或共同后继）→ 折叠为并行块；
3. 回边、条件边、自定义端口边 → 生成附着标注（引用目标的显示编号）；
4. foreach/subflow → 折叠块，body/子图递归渲染；
5. 输出中间结构（行列表 + 缩进 + 标注），前端渲染 HTML，同一结构序列化为纯文本供生成端点使用。

算法放服务端（新端点 `GET /api/workflow/outline`）还是前端 JS 二选一；倾向**前端**——cfg 已完整下发，画布渲染器已在前端，避免往返。纯文本序列化则服务端也要有一份（生成 prompt 用），可用同一套逻辑的 Python 实现（`workflow_outline.py`，输入 cfg 输出行结构，前后端各自渲染）。

### 10.2 复用清单

| 能力 | 现有落点 |
|---|---|
| cfg 读取/校验 | `read_workflow_config` / `workflow_execution_errors` |
| 回边识别 | `workflow_graph`（back edges） |
| 拓扑/入口/终点 | `workflow_entry_steps` / `workflow_terminal_steps` |
| 运行状态 | transitions 投影 + `node_runs`（attempt/耗时） |
| 步骤保存 | 现有 `POST /api/workflow`（面板编辑仍全量保存） |
| 审批/重跑/确认操作 | 现有 approval / rerun / skip / confirm API |
| JSONLogic 人话渲染 | 新增小工具函数（有限操作符集,orbit-safe-v1 profile 只有比较/布尔/var/长度/空值,穷举可译） |

### 10.3 分阶段

| 阶段 | 内容 | 依赖 |
|---|---|---|
| L1 | 只读列表视图 + 视图切换（渲染算法 + 行规范） | — |
| L2 | 详情面板（步骤/边编辑，替代双击弹窗） | L1 |
| L3 | 运行时状态叠加 | L1 |
| L4 | 行级 diff 预览（对接对话修改） | L1 + authoring 阶段 B |
| L5 | 纯文本序列化进生成 prompt | L1 + authoring 阶段 A |

L1-L3 不依赖 authoring 方案,可独立先行;与 authoring 阶段 C（标准形状）互斥取舍——列表视图落地后,画布形状美化优先级自然下降,建议 C 降级或砍掉。

## 11. 风险与边界

| 风险 | 对策 |
|---|---|
| 复杂图列表表达不了（交叉 join、非平面） | 局部降级提示 + 画布兜底（§9）;缩进上限 3 层硬约束 |
| 编号漂移（增删步骤后"第 3 步"含义变化） | 编号只在渲染层,数据层永远用 step.id;对话修改的编号解析基于**发起时的快照**,应用前校验目标仍存在（对齐 authoring §15 preconditions 思路） |
| 并行块识别歧义（非对称分支、部分汇合） | 识别失败的子图整体降级为"复杂区块 [画布查看]",不猜 |
| JSONLogic 人话翻译覆盖不全 | orbit-safe-v1 操作符集有限,穷举可译;未来扩展操作符时同步扩展翻译,漏译显示原始表达式 |
| 两视图编辑冲突 | 同一份 cfg、同一个保存端点,无并行状态;切视图即重渲染 |

## 12. 总结

列表视图把"顺序 + 例外"的流程本质直接映射为"编号大纲 + 附着标注"，覆盖 orbit 90% 的工作流形态；prompt 编辑从弹窗改为主从式侧栏面板;运行状态、diff、对话修改都在行级获得比画布更成熟的交互范式。画布保留为高级视图兜底非平面结构。数据层、引擎、校验零改动——这是一个纯展示/交互层的演进，L1 只读视图即可独立交付验证价值。

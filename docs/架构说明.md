# Orbit 架构说明

> 本文描述当前代码库的运行结构、主要模块和任务从 Goal 到完成的完整路径。

## 1. 项目定位

Orbit 是一个本地运行的多 Agent 工作流编排器。用户把一个 Goal 交给 Orbit；Orbit 按可配置的工作流图把它推进到不同步骤，并由对应的 Agent CLI 执行实际工作。默认工作流在设计阶段由 Goal 自身运行一次，随后由 Decompose 拆出工作项，再让各工作项独立完成实现、评审、测试与集成。

系统使用 Python、Starlette、uvicorn 和 SQLite；Web UI、调度器与内嵌 Runner 可以在一个进程中启动，也可拆分成一个服务进程和多个 Runner 进程。

```text
浏览器
  │  HTTP（仅本机）
  ▼
Starlette / Web UI / API
  │                 │
  │                 ├── Scheduler：推进工作流
  │                 ├── Watchdog：超时、健康检查、收敛验证
  │                 └── Embedded Runner（可选）
  ▼
SQLite Store  ←──────── 独立 Runner 进程也连接同一数据库
  │
  ├── run_jobs 队列
  ├── tasks / transitions / task_runs
  └── workflow_actions
  │
  ▼
Agent CLI（在任务 worktree 或项目根执行）
```

## 2. 启动与部署形态

命令行入口在 `src/orbit/__main__.py`。

| 命令 | 功能 |
|---|---|
| `orbit start` | 零配置启动。将项目状态目录加入 `.gitignore` 后启动服务，不会把默认工作流复制到项目。 |
| `orbit serve` | 启动 UI、HTTP API、Scheduler 与默认的内嵌 Runner。 |
| `orbit config` | 将默认工作流实体化为项目中的 `.orbit/workflow.json`，用于修改和提交。 |
| `orbit runner` | 启动独立 Runner，只从队列认领并执行工作。 |

默认地址为 `127.0.0.1:8848`，`start` 与 `serve` 都支持 `--host`、`--port`、`--db`、`--no-runner` 与 `--runner-concurrency`。

常见部署方式：

1. **一体模式**：`orbit serve`。适合本机快速使用，服务重启会中断内嵌 Runner 中正在执行的进程。
2. **解耦模式**：`orbit serve --no-runner` 加一个或多个 `orbit runner`。服务负责 UI、调度和恢复；Runner 可按 Agent 或步骤筛选任务，也可放在不同机器上共享同一数据库。

## 3. 源码与职责地图

| 路径 | 主要职责 |
|---|---|
| `src/orbit/__main__.py` | CLI 参数、项目初始化、服务和独立 Runner 的启动。 |
| `src/orbit/server.py` | 工作流引擎、调度器、Runner 执行、进程控制、Git worktree、HTTP API、Goal 收敛验证、健康检查。 |
| `src/orbit/store.py` | SQLite schema、迁移、事务性读写、任务/运行/队列/租约数据访问。 |
| `src/orbit/project_index.py` | 本机多个 Orbit daemon 的索引、探活与项目切换。 |
| `src/orbit/static/ui.html` | 单文件 Web UI：工作流编辑、Goal/任务面板、运行日志、恢复操作。 |
| `src/orbit/static/vendor/dagre.min.js` | 工作流图的自动布局。 |
| `tests/` | Store、工作流、Runner/工作树、项目索引和前端资源的回归测试。 |

## 4. 核心数据模型

`Store` 是 SQLite 的唯一访问层。数据库按项目隔离，默认位于用户目录下的 Orbit 项目数据库目录；项目内 `.orbit/`（兼容旧 `.dev_loop/`）保存工作流配置、设置、运行日志与 worktree。

主要数据概念如下：

| 数据 | 用途 |
|---|---|
| `agents` | 已注册 Agent 的名称、描述与最近活动时间。 |
| `messages` | 原始消息和任务消息，用于兼容任务协作模型。 |
| `tasks` | Goal、业务子任务和步骤卡片；保存状态、父子关系、步骤、预算、结构化结果等。 |
| `task_transitions` | 工作流步骤的派发、完成、返工、阻塞、跳过等审计记录。 |
| `task_runs` | 每次 CLI 或验证命令的执行记录：命令、退出码、token、PID、日志目录、起止时间。 |
| `run_jobs` | Runner 消费的持久化队列，带认领租约和心跳。 |
| `workflow_actions` | Scheduler/Watchdog 的幂等动作，例如派发恢复与 `goal_verify`。 |

其中 `task_transitions` 是工作流真实状态的审计来源；任务表中的可见状态是面向 UI 的投影。这样可以在返工、并行分支和崩溃恢复后重建活动步骤。

## 5. 工作流配置与执行语义

工作流配置由步骤（`steps`）和边（`edges`）组成。每个步骤可配置：

- Agent 列表与每个 Agent 的命令覆盖；未覆盖时使用该 Agent 的内置 CLI 命令。
- Prompt、是否必经（`required`）、是否隔离（`isolate`）、超时、验证命令等执行属性。
- 特殊标记：`decompose` 用于 Goal 拆分，`integrate` 用于合并工作树。

默认流程为：

```text
intake → product_design → ui_design → architecture → decompose
        → implement → review → test → integrate
                         ↖────── rework ──────┘
```

`decompose` 是 Goal 与工作项的边界：此前的设计步骤在 Goal 上执行一次；它成功后创建一个或多个业务子任务，子任务从其后继步骤开始。没有 `decompose` 标记时，Goal 本身直接走完整流程，不创建子任务。

工作流写入（保存、重置默认）在存在已启动但未终结的 Goal 时被锁定，包括 `blocked` 与 `stalled`，避免删除仍需重跑的旧步骤定义。

## 6. 一次步骤执行的生命周期

```text
1. Engine 根据图和前置条件决定可派发步骤
2. Scheduler/Engine 创建 run_job，并记录 dispatched transition
3. Runner 认领 job，创建 task_run，准备工作目录与日志
4. Runner 启动 Agent CLI，持续写 stdout/stderr
5. Runner 解析结果、token 与结构化输出，完成 run_job
6. Scheduler 单属主消费 finished job，推进、返工、阻塞或关闭任务
```

### Scheduler

`scheduler_tick` 是完成 job 的单属主消费者。它通过 SQLite 中的认领/应用租约处理完成结果，将并发 Runner 的输出串行应用到工作流，避免两个 Runner 同时推进同一任务。

### Runner

`runner_loop` 可以有多个实例和多个并发 worker。它从 `run_jobs` 认领符合范围的任务，启动 Agent CLI，捕获 stdout/stderr，写入运行日志，并把 outcome/result 回报给队列。Runner 只执行；工作流推进由 Scheduler 完成。

### Prompt、结果与 token

普通步骤的引擎 Prompt 要求 Agent 给出 `WORKFLOW_OUTCOME`、结果摘要、产物列表，并在能获得真实数据时输出 `TOKENS_USED`。系统优先解析可识别的 CLI 原生用量，再回退到自报值。

Decompose 必须输出一个严格 JSON 对象，因此 token 放在可选的 JSON 字段 `tokens_used` 中，而不是 JSON 外的文本行。`goal_verify` 是本地 shell 验证命令，不经过 Agent，不计入 token。

## 7. Git worktree 与验证

需要隔离的步骤会在项目状态目录下为任务创建独立 Git worktree 与任务分支，避免并行 Agent 修改同一工作目录。`integrate` 在主工作树完成合并与最终验收；无法使用 Git 时系统会降级为在项目根执行。

验证有两层：

1. **步骤 verify**：步骤可配置 shell 验证命令，在该步骤的工作目录运行；失败可推翻 Agent 自报的成功结果。
2. **Goal verify**：当所有工作项完成后，在集成后的主工作树运行 Goal 的自定义或自动检测验证命令。成功将 Goal 标为 `accepted`；失败标为 `stalled`，记录一次运行并通知 hub。修复相关子任务并再次关闭后会重新触发收敛验证，不会自动从头重跑整条工作流。

## 8. 恢复与可观测性

服务进程包含多个后台循环：Scheduler、步骤超时检查、任务健康检查、Hub 巡检、Goal 验证和已完成工作树清理。

- **租约与心跳**：Runner 认领 job 后周期续租；崩溃后的租约可被其他 Runner 回收。
- **健康检查**：检测死掉的 Runner、推进中断和未派发的返工目标，能自动恢复的会重新派发，不能恢复的会通知 hub。
- **Check & recover**：Execution 页面可立即触发同一套恢复检查，适用于重启后或人工排障。
- **超时与进程树控制**：Runner 支持软超时巡检与硬超时终止，按进程树清理子进程。
- **后台错误**：后台循环异常会写入服务日志，并通过 `GET /api/status` 的 `background_errors` 暴露失败次数、最近时间和摘要。

Hub 应通过 Goal/任务详情中的步骤卡片、Runs（stdout、stderr、result、verify）与通知文本定位问题，然后按需重跑、重新实现、跳过非结构性门禁或调整预算。

## 9. Web UI 与 HTTP API

Web UI 由 `/ui` 提供，所有数据操作经 `/api/*` JSON API 完成。主要页面包括：

- **Workflow**：编辑步骤、Agent、命令、Prompt、边和布局；可恢复默认流程。
- **Goal Execution / Board**：展示当前 Goal、步骤流、子任务和恢复操作。
- **Goals**：创建 Goal、配置预算/收敛验证、查看聚合进度和关闭 Goal。
- **Jobs / Runs**：查看队列认领状态、命令、stdout、stderr、结构化结果与 token。
- **Agents / Settings**：查看检测到的 CLI、配置工作流相关运行参数。

API 是本地控制面：服务同时校验对端 IP、Host 与浏览器 Origin，拒绝非 loopback 请求，降低本地 Agent 控制接口被跨站或 DNS 重绑定滥用的风险。

## 10. 多项目模型

每个项目有独立数据库和项目状态目录。`project_index.py` 在用户目录维护本机 daemon 索引；UI 通过 `/api/projects` 显示在线项目并切换查看。切换只改变前端目标服务，所有写入仍进入被选中项目自己的数据库与工作区。

## 11. 阅读和扩展建议

1. 先读 `__main__.py`，理解服务和 Runner 如何装配。
2. 阅读 `store.py` 的表结构与队列租约方法，理解持久化边界。
3. 阅读 `server.py` 中工作流图、派发、`scheduler_tick`、`run_queued_job` 和 `goal_verify_sweep`，理解完整执行链路。
4. 最后查看 `ui.html`，它展示了后端 API 在实际操作中的组合方式。

新增 Agent 时优先扩展 Agent 工具检测和命令解析；新增步骤语义时同时考虑配置归一化、Prompt 契约、工作流图校验、恢复逻辑和对应测试。
